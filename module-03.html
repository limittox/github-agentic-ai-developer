Unit 1 · Introduction

Agents run on tools, APIs, and boundaries

Modern agents rely on tools, APIs, and controlled execution environments to do meaningful work. In the GitHub ecosystem that means workflows, repositories, APIs, and external systems — all governed by permissions and execution boundaries. As agents get more autonomous (creating PRs, triggering workflows, touching infrastructure), it becomes critical to define how they operate, what they can access, and where they execute.

New concept

GitHub Agentic Workflows are a newer form of repository automation: you describe outcomes in Markdown and execute them through coding agents in GitHub Actions with strong guardrails. They augment traditional CI/CD, not replace it — GitHub calls the pattern "Continuous AI."

Unit 2 · APIs & workflows

How agents interact with GitHub

APIs — every action authenticated & auditable

GitHub's APIs let agents create branches and commits, read repository data, open/update PRs, and trigger workflows. Every request is authenticated — via personal access tokens, GitHub App tokens, or the workflow-provided GITHUB_TOKEN — so every agent action is permission-controlled and auditable.

Agents change code like developers do

  1. Select a base branch
  2. Create a new working branch
  3. Modify or create files
  4. Commit changes
  5. Open a pull request

Each step is a separate API operation (Git refs, repo contents, PRs) — agent actions stay fully aligned with GitHub's standard development model.

Actions is the execution layer

Agents don't execute on your machine. GitHub provides execution environments through workflows powered by GitHub Actions — running tests, validating changes, executing automation, deploying. The Copilot cloud agent itself runs in an Actions-powered environment.

Traditional vs agentic workflows

Traditional workflow

  • Deterministic, YAML-defined
  • You explicitly specify every step, trigger, condition
  • The workflow does exactly what's written

Agentic workflow

  • Markdown describes the outcome in natural language
  • Frontmatter defines boundaries: triggers, permissions, tools, safe outputs
  • Compiled into a lock file that Actions executes
  • Agent decides how to achieve the goal within constraints
  • Suited to open-ended but bounded tasks: triage, reporting, docs maintenance, CI-failure analysis, code improvement
agentic workflow — intent + guardrails 
---
on:
  schedule: daily
permissions:
  contents: read
  issues: read
  pull-requests: read
safe-outputs:
  create-issue:
    title-prefix: "[repo-status] "
    labels: [report]
tools:
  github:
---
# Daily Repository Status Report
Create a daily report for maintainers. Include:
- Recent activity (issues, PRs, commits)
- Key highlights and risks
- Recommended next steps

The agent interprets the intent and produces structured outputs, applied through controlled, reviewable steps — Markdown expresses intent, frontmatter defines boundaries.

End-to-end agent execution flow

  1. Task assigned — issue, chat, or CLI
  2. Agent selects repository and base branch
  3. Agent analyzes the codebase and plans changes
  4. API operations create branches and commits
  5. A pull request is opened
  6. Workflows run to validate (or deploy)
  7. Human reviews, approves, or requests updates

Sessions are observable and interactive — you can monitor progress through the session log, see actions as they happen, give feedback mid-task, and review the final PR. The result: all agent activity is scoped to the repository, controlled by permissions, executed through workflows, and visible for review.

Unit 3 · MCP

MCP servers, registries, and allow lists

MCP expands what agents can access — which is exactly why it needs governance. Three concepts solve three different parts of the same problem:

Server

Exposes tools to an AI client — sits between the client and the underlying system, presents tools, accepts requests, performs the real action.

Registry

Makes servers discoverable and trustable — a catalog so developers browse, select, install, and trust servers instead of hand-editing configs.

Allow list

Decides which servers are permitted — org/enterprise policy tied to the Copilot seat that governs the user.

Local vs remote servers

  • Local — runs on your machine; tighter config control, access to local resources; typically no external auth
  • Remote — hosted, accessed over the network; less setup, consistent across environments — recommended for most users
  • The GitHub MCP server (https://api.githubcopilot.com/mcp/) connects AI clients to repos, issues, and PRs; added via Copilot Chat → Agent mode → Tools → Add MCP server

Registries in practice

Supported IDEs use the GitHub MCP Registry by default; organizations can host a custom registry if it follows the MCP registry v0.1 specification:

  • Required HTTPS endpoints: GET /v0.1/servers, …/versions/latest, …/versions/{version}
  • CORS headers so Copilot can reach it (Access-Control-Allow-Origin: *, GET/OPTIONS)
  • Options: fork/self-host the open-source registry, run it in Docker, build your own — or use Azure API Center as a managed registry (base URL only, anonymous access enabled)

Allow lists — the enforcement layer

Configured at enterprise or organization level (AI controls → MCP, or Settings → Copilot → Policies). The policy decides whether developers can use MCP at all, and then:

ModeEffect
Allow allNo restrictions — developers can add and use any MCP server
Registry onlyOnly servers defined in the configured registry are allowed — even local servers must be listed, with exactly matching server IDs

Once selected, the policy applies immediately to all developers.

Module 02 echo

More tools means more responsibility: expanding MCP access increases blast radius. Safe MCP at scale needs all three — a server that exposes tools correctly, a registry that makes approved servers discoverable, and an allow list that limits what can actually be used.

Unit 4 · Boundaries

Execution context and boundaries

Execution context defines where agents operate. Four boundary layers combine:

1 · Repository scope

The Copilot cloud agent is explicitly enabled per repository (Settings → Copilot → Cloud agent). Once configured, the agent is scoped to that repository and can't operate outside it.

2 · Custom agent scope within the repo

Custom agents (e.g. .github/agents/security-reviewer.agent.md) refine scope further:

.github/agents/security-reviewer.agent.md 
applyTo:
  - '**/*.js'
  - 'src/auth/**'
tools:
  - read_file
  - search_files
  • Custom agents define: which files to focus on (applyTo), which actions they can perform (tools), how to behave (instructions)
  • Custom agents don't define: branch creation, PR behavior, execution isolation — those are enforced by the execution context, not the agent file

3 · Branch-based isolation

Agents never work directly on main: they create a branch off your selected base branch, change files there, and open a PR targeting the base. All modifications go through review before merge — a key safety mechanism.

4 · Workflow + permission boundaries

Workflows are controlled execution containers — clean environments, repeatable runs, captured logs. Permissions via GITHUB_TOKEN allow or restrict reading contents, writing code, creating PRs, accessing secrets, triggering workflows — always explicit, always minimized.

invoking an agent inside CI — the workflow is the boundary 
on:
  workflow_dispatch:
  schedule:
    - cron: '0 9 * * *'

permissions:
  contents: read

jobs:
  agent-task:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '18' }
      - env:
          COPILOT_GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          npx @github/copilot-cli --agent security-reviewer \
            -p "Review code vulnerabilities" --no-ask-user

Defense in depth — agentic workflow guardrails

  • Read-only tokens by default — the agent inspects state without direct writes
  • Safe outputs — the agent proposes actions; a separate gated step decides what's allowed
  • Zero secrets in the agent process — credentials stay out of the coding agent's runtime
  • Sandboxed, containerized execution
  • Network isolation with allowlisted outbound access
  • Threat detection scans proposed outputs before any write is applied
Why boundaries matter

Repository scope + branch isolation + workflow execution + permission control = agents operating within clear, enforceable limits — preventing uncontrolled changes to production code, access to unintended resources, and unsafe or unreviewed execution. This mitigates over-privileged agents, prompt injection, and unintended repository changes.

Unit 5 · Limits & protections

Execution limits, safeguards, and resilience

Platform-level controls (repo/org settings) govern which workflows are allowed, what actions can run, and what GITHUB_TOKEN may do. On top of that, agent-driven workflows need their own resilience patterns:

SafeguardHow
Error handlingFail fast (continue-on-error: false), meaningful error logs, prevent partial/inconsistent changes
RetriesHandle transient failures — rerun failed steps, retry loops in scripts (for i in 1 2 3; do … && break; sleep 5; done)
RollbacksNaturally supported by branch isolation + PR review; also close/discard the PR, or revert commits if merged
EscalationRequired PR review, auto-assigned reviewers, workflow steps that notify maintainers — critical decisions stay human
TraceabilityWorkflow logs, commit history, PR discussions; clear commit messages, branch-scoped changes, everything reviewed via PR
Environment protectionsApprovals before jobs continue, restricted secret access, controlled deployment targets

The result: execution that is resilient (handles failure), controlled (prevents unsafe changes), auditable (all actions visible), and human-governed (escalation guarantees oversight).

The GITHUB_TOKEN loop-prevention quirk

Exam favorite

Most actions performed with GITHUB_TOKEN don't trigger additional workflow runs — deliberate protection against infinite automation loops. GitHub App tokens and PATs can trigger further runs, so workflows authenticated that way need careful design to avoid recursion.

Session visibility

Agent execution is visible while it runs: monitor logs, inspect actions, provide follow-up prompts to adjust behavior. You stay in control throughout.

Key takeaway

Agent execution is controlled through repository scope, branch isolation, workflow permissions, environment protections, and human decision points. Agents prepare changes — you remain responsible for reviewing and finalizing them.
