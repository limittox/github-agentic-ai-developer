Designing Agent Architecture & SDLC Integration

01
Unit 1 · Introduction

Architecture, not freedom

Agents can interpret goals, propose solutions, and act inside repositories — but capability alone doesn't make an agent reliable. Without well-defined architecture, agents act too early, produce unclear changes, or operate without validation. In production, those failures are real risks to code quality, security, and stability.

The thesis

Designing an agent system in GitHub isn't about giving the agent more freedom. It's about defining how work flows through the system using enforceable GitHub functions — pull requests, workflows, repository rules — so every action leaves a visible record, is validated by objective signals, and is accepted only when it meets policy.

Glossary distinctions to hold onto

  • Plan (artifact) — a structured description of intended changes, typically in the PR description: goal, scope, steps, risks, validation criteria
  • Planning (agent capability) — the process of generating/refining a plan from a task, issue, or user input; can start from a GitHub issue or the Agents interface
Scope note

This module covers repository-level architecture (PRs, checks, rules). Production agent systems also add environment-level controls — e.g. Copilot cloud agent's configurable firewall limiting external network access. Those define what the agent can reach at runtime; PR governance defines what changes get accepted.

02
Unit 2 · Responsibilities

Map agent responsibilities to the SDLC

An agent treated like a general-purpose developer is hard to reason about, limit, or audit. Instead, map the agent to specific lifecycle stages where GitHub can enforce boundaries. Most teams start with implementation and validation, where PRs and workflows provide natural control points.

SDLC stageTypical agent responsibilityPrimary artifact
PlanningDraft scope, plan steps, define success criteriaIssues, PR descriptions/comments, Agents tab
ImplementationCreate branch, make changes, open/update PRBranch, commits, pull request
ValidationRun checks, attach artifacts, iterate on failuresWorkflow runs, checks, artifacts
DeploymentUsually restricted; approvals for sensitive actionsEnvironments & deployment approvals

Architectural boundaries that reduce blast radius

  • Scope early — limit which directories the agent can modify, by policy and ownership
  • Treat workflow and infra changes as higher risk than application code
  • Prefer PR-based work even for automation — never direct-to-default-branch
The design boundary

Agents propose; humans and policy accept. The agent prepares work and submits it through a PR — repository policy and human reviewers decide whether it merges or deploys.

Example: a dependency-remediation agent scoped to implementation detects a vulnerable dependency → creates a branch → updates dependency + lockfile → opens a PR with a structured plan and expected success signals. Its scoped responsibility ends there — validation and acceptance happen through checks, reviews, and policy.

03
Unit 3 · Task contracts

Define inputs, outputs, and success criteria

Under-specified tasks produce changes that look plausible but don't solve the underlying problem. Every agent task should be a contract with three parts:

  • Inputs — what the agent needs: issue context, constraints, boundaries
  • Outputs — what it produces: plan + PR + evidence
  • Success criteria — how results are evaluated: checks, scans, review outcomes

Example contract: vulnerability remediation

PartContents
InputsSecurity alert / issue link · scope: src/ and dependency files allowed, infra/ excluded · constraints: no workflow changes without platform review, no secrets, no direct-to-main pushes
OutputsA PR containing a structured plan, a bounded changeset on an agent branch, and evidence links to workflow runs
Success criteriaRequired checks pass · security signal actually resolved · scope matches intent (no unexpected files) · rollback/escalation path recorded for higher-risk changes
Critical tip

"CI passed" is necessary but not always sufficient. Make success criteria reflect the real intent — "vulnerability resolved," not just "tests passed." An agent can update a dependency while the vulnerable version stays reachable through a transitive dependency.

Turning success criteria into enforcement

A CI workflow becomes an enforceable success signal when marked as a required status check:

.github/workflows — tests + CodeQL as required checks 
on:
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test

  security-analysis:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      security-events: write
    steps:
      - uses: actions/checkout@v4
      - uses: github/codeql-action/init@v3
      - uses: github/codeql-action/analyze@v3

Once required via rulesets/branch protection, the PR cannot merge until the check passes — success is enforced by the system, not assumed by the agent.

04
Unit 4 · Separation of concerns

Separate planning, reasoning, and execution

Reliable agent systems separate planning (what and why), execution (concrete repo changes), and validation (evidence that outcomes meet criteria). When planning and execution blur together, reviewers only see the final diff — they lose the chance to validate intent early, catch misunderstandings, and control scope before impact.

GitHub maps this naturally: planning → PR description / issue / template artifact; execution → commits on a branch; validation → checks, scans, artifacts, reviews.

The key design choice: when does validation happen?

Option A · Plan-first PR

  • PR opens containing only the plan — no code yet
  • Reviewers discuss, refine, approve the plan in the PR
  • Only after approval does the agent implement
  • Validation of intent happens before any code exists
  • Best for high-risk / hard-to-reverse work: workflows, infra, auth, production

Option B · Plan + execution together

  • PR contains the plan and initial commits at once
  • Faster iteration; plan visible alongside changes
  • Code exists before the plan is fully validated
  • Risk lives in the proposal stage — merge is still gated
  • Best for low/medium-risk, easily reversible changes
Key takeaway

The choice is never whether work is reviewed — it always is. The choice is when the system allows code to be generated relative to human validation. Separation creates the opportunity to review intent before accepting impact.

Enforcing planning boundaries

  • Capability boundary — planning agents get read-only tools; they physically can't modify files
  • Explicit handoff — execution starts only after plan approval, via a deliberate transition to an implementation agent
  • Tool gating — orchestrators run planning with tools disabled, enabling them only once the plan is accepted
  • Plan-mode workflows — interfaces that generate a plan artifact and pause before applying changes
Guidance vs enforcement

Treat "instructions not to edit" as guidance; treat tool allowlists and gates as enforcement. Only the second category is guaranteed to hold.

05
Unit 5 · PR governance

Pull requests are architectural control points

PRs aren't just collaboration tools — they're enforcement mechanisms. Well-designed architectures route all agent changes through PRs and enforce merge requirements by policy:

Agent creates branch
↓
Opens PR — includes plan
↓
Required reviews validate approach
↓
Actions run required checks
↓
Checks pass + approvals complete → merge allowed

PR template — consistent plan & evidence

.github/pull_request_template.md 
## Plan (required)
- **Goal:**
- **Scope (paths/files):**
- **Steps:** 1. / 2. / 3.
- **Success criteria (verifiable):**
  - [ ] Required checks pass
  - [ ] Security signals reviewed
- **Risks + mitigations:**
- **Rollback / escalation plan:**

## Evidence
- Workflow run(s):
- Scan results (if applicable):

## Review checklist
- [ ] Plan reviewed and approved
- [ ] Required reviews satisfied
- [ ] Required checks satisfied

Plan gate — expectation becomes guarantee

A "plan gate" workflow that fails when the plan artifact is missing, marked as a required status check, turns "include a plan" from a process expectation into a system guarantee — PRs physically can't merge without it.

CODEOWNERS — route by risk

CODEOWNERS 
/security/            @security-team
/.github/workflows/   @platform-team
/infra/               @platform-team
*                     @core-team

Combined with required review policies, changes touching high-risk paths can't merge without the right experts seeing them.

Watch for

If an agent can bypass required checks or merge without reviews, the architecture has lost its primary safety mechanism. That's a workflow design failure, not a model problem.

06
Unit 6 · Workflows

Reliable workflows — autonomy, outputs, triggers

Autonomy must be designed, not assumed

Express autonomy levels through policy, matched to task risk:

Task typeExample pathsRiskAutonomy design
Lowdocs/, formattingLowAuto-merge allowed after required checks (and reviews, if configured) pass
Mediumsrc/, dependency bumpsMediumPR + checks + at least one review
Highinfra/, .github/workflows/HighCODEOWNERS + multiple reviews + stricter rulesets
CriticalProduction deploys, secretsCriticalEnvironment approvals — agent prepares but can't execute

Environments are the strong control for the critical tier — a job targeting environment: production with required reviewers pauses until approval.

Outputs are workflow contracts

Data that downstream steps or jobs consume is a contract, not "just logs":

  • Step outputs — pass values between steps in the same job ($GITHUB_OUTPUT)
  • Job outputs — pass values across jobs via needs dependencies
  • Env vars — runtime configuration; shouldn't replace outputs for structured data flow
cross-job handoff 
jobs:
  plan:
    outputs:
      plan: ${{ steps.generate_plan.outputs.plan }}
    steps:
      - id: generate_plan
        run: echo "plan=..." >> "$GITHUB_OUTPUT"

  implement:
    needs: plan
    steps:
      - run: echo "Using plan: ${{ needs.plan.outputs.plan }}"

Contexts — the right tool per purpose

  • github.* — event metadata and runtime decisions: "what triggered this run?"
  • vars.* — centrally managed, reusable configuration values
  • env.* — job-level environment variables and runtime configuration

Defensive gating for PR-only behavior

Workflows often have multiple triggers (push, workflow_dispatch, schedule). Guard PR-specific steps with a job-level condition so they never run outside PR context:

job-level guard 
jobs:
  validate-pr:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test
Key takeaway

Workflow reliability improves when plans and signals are structured outputs, guarded by event-aware logic.

Unit 7 · Operations

Operate agents — observability, tools, MCP, secrets, hooks

Minimum observability set

Every meaningful agent action must produce visible artifacts — without them you can't review behavior, debug failures, or do post-hoc analysis:

  • A visible plan artifact (PR description or discussion)
  • A bounded PR + commit history
  • Workflow run links for required checks
  • Durable uploaded artifacts — logs, test reports, traces (actions/upload-artifact)
  • Review outcomes — approvals or changes requested

Evidence must be traceable to a specific workflow run and commit — so audits can answer "which run produced this artifact, and against what code state?" Link runs and artifacts in the PR's Evidence section.

Tool governance — three boundaries

  • Capability boundary — which tools are allowed; prefer allowlists. Read-only toolsets for planning/review agents; implementation tools only for execution agents
  • Visibility boundary — whether the agent is user-selectable in interactive UI
  • Delegation boundary — which subagents can be invoked and how handoffs occur

Treat changes to tool allowlists as governance-sensitive changes.

MCP servers — extend tools safely

  • Transport — some MCP servers are remote endpoints, others local processes
  • Auth — tokens injected at runtime through protected secret boundaries, never committed
  • Namespace control — enable a narrow tool subset, not broad wildcards
Operational rule

Adding or expanding MCP tools increases blast radius — review it like a high-risk dependency.

Secrets — keep them out of repo content

  • Never in instruction files, committed config, or plain-text workflow YAML
  • Use protected secret boundaries designed for runtime injection
  • Pass secrets only to components that need them; scope by environment
  • The agent's runtime has its own secret boundary — don't assume it inherits repository CI secrets

Hooks — guardrails independent of model reasoning

Hooks (config files, e.g. under .github/hooks/) run custom commands at specific points in agent execution:

pre-tool-use hook — block unsafe commands 
{
  "name": "block-high-risk-command",
  "trigger": "pre-tool-use",
  "run": "if [[ \"$TOOL\" == \"delete\" ]]; then echo 'Blocked unsafe command'; exit 1; fi"
}
  • Pre-action hooks — validate or block unsafe actions before execution
  • Post-action hooks — log tool usage, outputs, decisions for auditing
  • Error hooks — capture failures, trigger escalation or alerting
Why hooks matter

Hooks are enforceable control points that operate independently of the model's reasoning. Instead of relying on instructions, they ensure rules are always applied during execution.

Reliability assumes failure

  • Bounded retries — a failed required check → the agent may revise the branch and rerun
  • Escalation — the same required check fails twice → escalate to a human with: what failed, what was attempted, what evidence exists, suggested next step
  • Rollback readiness — high-risk changes carry rollback notes; deploy sensitive config from explicit refs (commit/tag), not "latest on a branch"
  • Least privilege — e.g. permissions: contents: read, pull-requests: write — read content, update PR context, nothing broader by default
