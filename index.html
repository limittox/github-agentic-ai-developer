Module 01 — Study Guide

Foundations of Agentic AI in GitHub

AI-assisted development is shifting from tools that suggest code to systems that take action inside the SDLC. This module builds the mental model: what makes a system agentic, how the plan → act → evaluate loop works, and how GitHub keeps agent work safe, supervised, and traceable.

01
Unit 1 · Introduction

The developer's role is changing

Instead of tools that only suggest code, we now have systems that can act inside the software development lifecycle. GitHub Copilot's cloud agent already demonstrates this: it can explore a repository, propose a plan, make changes on a branch, and open a pull request for you to review.

As these systems become more capable, you're no longer just writing code — you're guiding, supervising, and validating systems that can plan, act, and improve over time. Agent workflows follow a plan → act → evaluate loop that refines each step against system feedback until the outcome meets the required standard.

What this module covers

  • Define agentic AI in the SDLC and distinguish agents from assistants
  • Explain and apply the plan → act → evaluate lifecycle
  • Describe GitHub as both system of record and control plane
  • Identify responsibilities, risks, anti-patterns, and traceability requirements
  • Apply the contributor model to evaluate agent-generated work
Heads up

Some controls in this module (rulesets, branch protection, required checks) must be configured by repo or org administrators. The supervision model applies everywhere — enforcement needs the right permissions.

02
Unit 2 · Define agentic AI in the SDLC

Assistant vs. agent — the line that matters

An assistant responds to a prompt, generates output, and hands control back to you. An agent goes further: it interprets a goal, decides on intermediate steps, uses tools, and takes action inside a workflow. That difference turns AI from something that helps with development into something that participates in development.

Assistant · reactive

  • Waits for you to decide what happens next
  • Suggests code, explains output, summarizes changes
  • Takes no repository actions
  • You apply every step manually

Agent · goal-driven

  • Maintains a goal across multiple steps
  • Decides intermediate actions on its own
  • Uses tools — GitHub API, CI workflows, repo writes
  • Creates durable artifacts: branches, commits, PRs
  • Iterates based on feedback — checks, reviews, scans

How agent behavior shows up in GitHub

Agent behavior is visible through the same structures developers already use:

  • Branches & commits — what changed
  • Pull requests — what is proposed, why, and for review
  • Workflows & checks — what evidence exists
  • Review comments & approvals — what humans accepted or rejected
Key idea

An agent does not replace the workflow. It enters the workflow as a participant — typically proposing changes on a branch and opening a PR that waits for review and validation before merge.

Worked example

A security alert is filed. Agent behavior: creates branch agent/bump-dep-2026-04-03, updates the dependency and lockfile, opens a PR with a summary and plan, then waits on CI and review feedback — revising if needed. Assistant behavior: you ask "how do I safely update this dependency?" and receive commands, a risk checklist, and suggested changes — but you create the branch and PR yourself.

Unit 3 · The agent lifecycle

Plan → Act → Evaluate is a loop, not a line

Agentic systems don't make one decision and stop — they operate in cycles. The agent repeatedly plans, acts, and evaluates until the task meets defined success criteria, or the work is escalated to a human.

→
Actbounded execution
→
Evaluatesystem signals
loops until success criteria met — or human handoff

Plan

The agent interprets the goal and determines the steps. In high-quality systems, plans are not hidden internal state — they're structured, reviewable artifacts: a plan section in the PR description, or a linked issue/checklist with scope and success criteria. Good plans include scope (what will change), success criteria (how you'll know it worked), and a rollback or escalation path.

Act

The agent executes in the repository: creating a branch, changing files, pushing commits, opening or updating a PR, and responding to review feedback. Execution stays bounded — on a branch, through PR workflows, never as uncontrolled writes to the default branch.

Evaluate

Evaluation is grounded in system signals, not the agent's confidence:

  • Workflow runs and status checks — build / test / lint
  • Code review feedback — requested changes, approvals
  • Security signals — code scanning (SARIF), secret scanning alerts, dependency alerts, push protection

When rulesets or branch protection require checks to pass, evaluation becomes an enforceable gate rather than an informal suggestion. If checks fail or risks remain, the loop continues: revise the plan, adjust actions, re-evaluate — or hand off to a human.

Quality bar

A high-quality agent system makes every phase visible: the plan is inspectable, action is bounded to repository workflows, and evaluation uses objective signals. Remove any piece and trust degrades.

04
Unit 4 · GitHub's role

System of record, and control plane

Agentic systems need an environment that can capture intent, record actions, enforce validation, and apply policy. GitHub plays two distinct roles at once.

System of record — what happened

GitHub stores the artifacts through which work is proposed and evaluated. In agentic workflows these do double duty: they support development and make agent behavior inspectable after the fact.

  • Repositories and branches
  • Commits and pull requests
  • Issues and discussions — context and intent
  • Workflow runs and artifacts — evidence
  • Review history — decisions

Control plane — what's allowed

GitHub controlWhat it enforcesWhy it matters for agents
Pull requestsChanges are proposed before mergingMakes agent work reviewable and discussable
Required reviewsHuman approval gatePrevents unreviewed merges; supports accountability
Required status checksCI evidence before mergingConverts evaluation into enforceable policy
CODEOWNERSReview routing by pathRight experts supervise high-impact changes
Rulesets / branch protectionCentralized branch policyPrevents unsafe merges; consistent guardrails
EnvironmentsApprovals for deployments / secretsControls sensitive execution and secret access

Least privilege in Actions

Permissions matter as much as checks. For agents, "what the agent can do" often reduces to "what the workflow token and tool credentials can do."

  • Set default GITHUB_TOKEN permissions conservatively — read-only where possible
  • Grant higher permissions only to the jobs that need them
  • Use environments with required reviewers to gate secrets and deployments
  • A human "Approve and run workflows" step guards privileged runs on untrusted changes
Remember

Enforcement depends on configuration and permissions — enabling required checks and rulesets is typically an admin task. The supervision model works everywhere; enforcement requires the controls to be turned on.

05
Unit 5 · Responsibilities & risks

Responsibility does not move with execution

Agents may execute work, but humans remain accountable for outcomes and the controls that govern execution. The accountable parties are the people who defined the task, set permissions, configured controls, and approved the change. The PR model makes this explicit: the system can propose; humans decide what is accepted.

Common anti-patterns → GitHub mitigations

Risk / anti-patternWhat it looks likeMitigation
Planless executionPR has a diff but no plan or rationaleRequire a plan section via PR template; required review before merge
Over-permissioned agentsWorkflows write broadly, access secrets freelyLeast-privilege GITHUB_TOKEN; environments with required reviewers; restrict workflow triggers
Hidden reasoningNo assumptions, scope, or decision trail — only the diffRequire plan; link workflow runs; record decisions in PR comments
Blind trust in automation"CI passed, ship it" mindsetCombine checks with CODEOWNERS, required reviews, and risk-based approvals
Careful

Checks only validate what they're designed to detect. A passing build does not mean the change is complete, appropriate, or low-risk.

Minimum audit trail for an agent contribution

  • A stated goal — issue link or PR description
  • An inspectable plan — PR plan section or file
  • A bounded changeset — branch and commits
  • Automated evidence — workflow runs and artifacts
  • Human judgment — review and approval
  • A clear outcome — merge, revert, or escalation

The goal isn't only compliance — it's operational understanding. When something fails, you need to reconstruct what changed, who approved it, what evidence existed, and what happened next.

06
Unit 6 · The contributor model

Judge the work, not the author

The most reliable way to evaluate agent output: stop treating it as categorically different. Treat it as a contribution. A pull request should answer the same questions whether its author is a human or an agent. This avoids two opposite errors: excessive suspicion ("reject it because AI wrote it") and excessive trust ("accept it because automation produced it").

Review rubric for agent PRs

  • Intent — clear goal and a visible plan?
  • Scope — files changed align with the plan?
  • Evidence — required checks pass; logs/artifacts available?
  • Ownership — CODEOWNERS reviewed sensitive areas?
  • Policy — complies with rulesets, branch rules, environments?
  • Fallback — rollback or escalation clear for high-risk changes?

What good looks like

A well-supervised agent contribution is:

  • Understandable — clear goal and plan
  • Bounded — scoped changeset, least privilege
  • Reviewable — right owners involved, evidence present
  • Policy-compliant — rulesets, branch rules, environments respected
  • Reconstructable — audit trail supports post-hoc analysis
The standard

This is not a special standard for AI. It's the standard of a healthy engineering workflow applied consistently — grounded in PRs, checks, reviews, policy, and human judgment rather than hype or fear.
